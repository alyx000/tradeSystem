# 实施计划与测试验证（默认强制）

## 适用范围

凡使用 **Plan mode**、**CreatePlan**，或向用户交付**实施计划 / 改造方案**（含架构、存储、多文件 CLI/API 变更），均视为「实施计划」。

## 默认要求（不必等用户说触发词）

1. **第一版计划即须包含「测试验证方案」**，与实现清单并列，不得事后才补（纯文案/单文件注释除外）。
2. 方案至少写明：
   - **分层**：数据层 / 业务或采集 / CLI 或 API / 可选 UI 中，本次涉及哪些层；
   - **验收命令**：具体 `pytest` 路径（或 `scripts/tests/` 下模块）+ 是否跑 `make check-scripts` / `make check`；
   - **完成标准**：哪些命令全绿、哪些 Warning 不可接受（与 [`dev-workflow.md`](dev-workflow.md) 一致）。
3. 若触发 [`test-design.md`](test-design.md) 中的条件（新模块、存储/协议变更、3+ 文件等），分层测试方案须**符合**该文件的金字塔与隔离原则。
4. 计划须列明主要改动项、文件范围、依赖关系与非目标，避免遗漏或越界。

## 不再规定固定 Agent 分工

本规则**不再要求**复杂任务输出多 Agent 并行分组，也不再规定：

- 固定的角色 / 执行 Agent / 专项关注 / 禁区 / 冲突标注表；
- 按主线、打断频率、时长选择 Claude Code、Codex BG 或其他执行体；
- 测试、前端、后端、文档必须交给特定 Agent；
- CreatePlan 后必须额外启动 subagent 做方案审查。

是否使用并行或额外 Agent，仅按当前执行环境、用户明确要求和任务实际需要决定，不作为实施计划的固定章节或强制门槛。

## 输出计划前的强制步骤

1. **确认范围**：列出主要改动项、涉及文件、依赖、非目标和已知风险。
2. **确认测试验证方案**：计划正文必须包含分层、验收命令和完成标准。
3. **确认阶段审查点**：多阶段实施时，每个实质性阶段完成并通过阶段测试后，立即执行适用的代码审查门，不把所有审查积压到最终收尾。

多阶段计划的每个阶段应写明：

```text
完成标准：阶段测试全绿；若本阶段构成实质性代码改动，立即按
code-review-gate.md 与 post-dev-codex-review.md 完成审查后再进入下一阶段。
```

## 与现有规则的关系

- [`dev-workflow.md`](dev-workflow.md)：开发前设计验证方案、开发后跑通 pytest。
- [`test-design.md`](test-design.md)：重大变更的分层测试结构设计。
- [`code-review-gate.md`](code-review-gate.md) 与 [`post-dev-codex-review.md`](post-dev-codex-review.md)：实质性代码改动后的代码审查门。

本规则只解决：在**计划文档阶段**明确范围与验证方案，避免“只写实现、后补测试”。

## 例外

- 仅改文档、注释、格式化、与行为无关的配置文案：可注明“无单测变更”，不写分层测试。
