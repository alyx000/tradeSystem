# 行为基线规则

> 本规则提供通用行为约束，用于降低 Agent 在实现、排查、review、文档同步中的常见失误。
> 业务红线、写入边界、CLI 语义、确认门槛仍以 `AGENTS.md` 与 `.cursor/agent-context/` 为准。

## 适用范围

适用于以下任务：

- 代码实现与重构
- Bug 排查与修复
- CLI / API / service 对齐
- 技术方案与执行计划
- 文档同步与规则调整

纯闲聊、简单翻译、一次性短问答可按常识简化执行。

## 1. 先校验假设，再开始动作

**不要替用户做隐式假设，不要把不确定当确定。**

- 遇到日期、范围、入口、真实数据来源不明确时，先查清仓库现状或显式说明假设。
- 需要判断系统行为时，优先核对真实 `CLI` / `API` / `service` / 测试，而不是凭记忆补全。
- 输出中必须区分 `[事实]` 与 `[判断]`，不得把推测包装成既成事实。
- 如果同一需求存在多种解释，不要静默选一种继续做。
- 如果发现更简单、更小的改法，应明确指出，不要默认走更重方案。

## 2. 简洁优先，复用现有结构

**只做解决当前问题所需的最小改动，不为未来猜测提前造抽象。**

- 不新增用户未要求的功能、配置项、扩展点或“灵活性”。
- 优先复用已有 `service`、现有命令组、既有状态流转和文档结构。
- 单次使用的逻辑不要提前抽象成框架级组件。
- 不为当前不存在的场景补写复杂兜底。
- 如果 50 行可完成，不要写成 200 行。

## 3. 精准修改，只清理自己的影响面

**每一处改动都应能直接追溯到当前任务。**

- 不顺手重构相邻模块，不“顺便优化”无关代码、注释或格式。
- 保持现有文件分层、命名和文档组织方式，除非本次任务明确要求调整。
- 如果改动导致索引、入口文档、引用关系失效，补齐这些由本次改动直接带来的同步项。
- 如果发现与本任务无关的死代码或设计问题，可以指出，但不要未经要求直接删改。
- 若工作树里存在他人的未提交改动，避免覆盖，除非任务要求明确涉及同一位置。

## 4. 目标驱动执行，结果必须可验证

**先定义完成标准，再实现，再验证。**

- 开始前先明确验证方式：跑什么命令、看什么结果、哪些失败不可接受。
- 修 Bug 时，优先先定位可复现证据，再修复，再回归验证。
- 做 CLI / API / service 语义调整时，验证默认值、校验、状态流转是否一致。
- 文档或规则改动若不涉及代码行为，应明确说明“本次无单测变更”，并完成结构性校验。
- 结束时必须给出验证结果，而不是只说“应该可以”。

## 与项目规则的关系

- `AGENTS.md` / `CLAUDE.md` / `.cursor/agent-context/`：定义业务边界、写入语义与主题入口。
- `dev-workflow.md`：定义开发前后验证与测试报告要求。
- `implementation-plan.md`：定义计划阶段的验证方案与并行分组要求。
- `test-design.md`：定义重大变更时的分层测试设计。

本规则只解决**行为层**问题：少猜、少做多余事、少误改、强验证闭环。
