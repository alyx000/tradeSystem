# 认知候选质量规则

本文件约束 **“提候选”** 阶段的输出质量，解决“能落库但过短、过像当期观点”的问题。

## 1. 默认目标

- 默认给 **更少、更硬、更可复用** 的候选
- 默认先做 **候选预览**，不是急着写库
- 默认输出 **强候选**，不是占位版短描述

如果一个点只能写成“某板块很强”“某公司有机会”“这方向还能涨”，通常还不该进认知层。

## 2. 强候选最小标准

每条候选至少要能回答下面 6 个问题：

| 项 | 必须回答的问题 |
|---|---|
| `title` | 这条认知的核心规律是什么？ |
| `description` | 它为什么成立？底层驱动是什么？ |
| `pattern` | 以后遇到类似情况，怎么复用？ |
| `applicability` | 它适用于什么盘面 / 周期 / 结构？ |
| `invalidation` | 什么情况下不能再套用？ |
| `why_keep` | 为什么这条值得长期保留，而不是留在老师观点里就够了？ |

### 强候选模板

```md
标题：

分类：

description：

pattern：

适用边界：

失效边界：

为什么值得保留：
```

## 3. 不建议落库的典型情形

出现以下任一情况，优先输出 **“不建议落库”**，并说明原因：

- 只是当期行情观点，没有稳定复用价值
- 只是个股结论，无法抽象成框架
- 只能描述“结果”，说不清“为什么成立”
- 适用边界和失效边界完全说不清
- 本质上是仓位建议、价格判断或情绪口号
- 与现有 active / candidate 高度重复，仅是措辞变化

## 4. 常见弱候选反例

下面这些通常不够格：

- `商业航天还能涨`
- `这家公司有问题`
- `市场要小心`
- `高位容易分歧`
- `资金去了低位`

问题在于：过泛、过短、没有边界、没有复用方法。

## 5. refine 默认动作

用户说下面这些话时，默认不要直接改库，先给 refine 候选：

- `太简单了`
- `这个描述不够好`
- `再精炼一点`
- `这条容易被误用`

默认流程：

1. 先列出 refine 候选文案
2. 明确指出改动点：补了什么边界、为什么更稳
3. 等用户确认后再执行 `cognition-refine`

## 6. 发给用户前的自检清单

发候选前至少自检一次：

- 这条是“可复用认知”，不是“当期观点”吗？
- 半年后再看，别人还能直接看懂吗？
- `description` 解释了为什么成立吗？
- `pattern` 真的能迁移到下一次吗？
- 写清了适用边界吗？
- 写清了失效边界吗？
- 如果不建议落库，我是否明确说了原因？
